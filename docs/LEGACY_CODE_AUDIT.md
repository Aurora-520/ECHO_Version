# 旧 MaixCAM 代码审计

## 1. 总体判断

旧代码是比赛原型，不是正式架构。其价值在于记录了实际尝试过的几何算法和任务流程；不能
直接复制到正式工程。正式工程只允许逐个提炼为可离线测试的纯函数。

## 2. 可复用算法思想

- Maix 图像与 OpenCV/NumPy 图像转换。
- 灰度化、Gaussian blur、自适应阈值/Otsu 和形态学闭运算。
- 轮廓面积、周长、圆形度、矩形度和凸包比筛选。
- `cv2.moments` 计算图形中心，`minEnclosingCircle` 精修圆心。
- `approxPolyDP` 动态 epsilon 提取三角形、矩形和十字角点。
- 角点围绕中心排序，并统一到图像中心坐标系。
- 相邻角点均匀插值，生成三角形、矩形、圆、星形和十字轨迹点。
- YOLO 粗检测框内使用传统视觉二次精修。

复用条件：移除设备访问、硬编码尺寸、串口输出和隐藏状态，为每个函数增加参数、坐标系、
边界条件、失败原因和固定样例测试。

## 3. 不可继承结构

- `version 4.4.0(1).py` 将相机、显示、模型、UART、算法、Mission 和无限循环混在 1725 行。
- 导入阶段立即打开相机/UART并加载两个模型，任一资源缺失都会阻止启动和离线测试。
- 使用全局变量和函数属性保存状态，重新进入任务时没有可靠 reset。
- UART 直接发送文本 `x,y`，命令为单字节字符，没有 CRC、序号、确认、时间戳和超时语义。
- `0,0` 同时表示未检测、扫描、待机和失败，也可能是合法居中坐标。
- 分辨率固定 `448x448`，中心固定 `224,224`，ROI、面积、阈值、偏移和完成计数散落代码。
- 日志打印、相机读取、模型推理和 UART 发送处于同一同步循环，执行时间无界。
- 没有录像、失败帧、配置版本、健康状态、FPS/延迟/drop 或离线回放。

## 4. 已发现具体缺陷

- `get_circle_points()` 使用未定义的 `pi`、`cos`、`sin`。
- 轨迹衔接中存在 `next_x =+ 5`，这会赋值而不是累加。
- 方形下一点补偿连续两次写 `next_x`，疑似第二次本应写 `next_y`。
- 注释中的连续帧数量与实际阈值多处不一致，例如显示 `/10` 但判断 `>=80`。
- 目标消失时周期性发送 `0,0`，MCU 无法区分过期、无目标和中心命中。
- 扫描期也发送 `0,0`，控制侧无法知道视觉仍在初始化。
- function attribute 状态在退出/重进任务时可能继承旧索引和完成状态。
- 模型标签顺序只通过硬编码 class_id 推测，没有标签文件证据。

## 5. 模型状态

`model_262872.mud` 和 `model_271670.mud` 仅在源码字符串中出现，资料目录中未找到模型、标签、
训练数据或说明。不得根据旧 `class_id` 硬编码结果推断模型真实标签。
