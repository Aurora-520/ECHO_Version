# 鲁棒性基础：算法不是在“物体”上工作，而是在像素上工作

## 1. 为什么同一个圆换个灯就可能识别不到

人看到的是“白纸上的黑圆”，算法看到的是一组数值。灯变暗、产生阴影、相机自动提高增益或
白平衡改变后，同一个位置的像素值会变化。固定写死 `gray < 80`，其实不是在找黑圆，而是在找
“当前相机和当前灯光下小于 80 的像素”。

因此真正稳定的算法通常会结合局部对比度、轮廓形状、面积、圆度、几何关系和多帧一致性，
而不是只靠一个阈值。

## 2. 鲁棒不等于永远返回结果

最危险的情况不是 `valid=false`，而是算法看不清时仍然自信地给出错误坐标。视觉系统应有三种
区域：

```text
正常范围 -> 输出有效结果
边界范围 -> 降低置信度、加强一致性检查
能力外部 -> valid=false + fault，等待恢复
```

MCU 只使用新鲜、有效并达到置信度要求的结果。

## 3. 自动曝光为什么既有帮助也有风险

自动曝光能在亮暗变化时调整画面，但调整需要时间。灯突然打开时，前几帧可能过曝，随后才
恢复。如果算法不知道曝光正在变化，就可能把瞬态亮斑当成目标。

调试时应同时保存原图、曝光、增益、白平衡和时间序列。最终选择锁定曝光还是受限自动曝光，
需要同场景 A/B 测试决定。

## 4. 为什么要有负样本

只收集“有圆的图片”只能测漏检，不能测算法是否把螺丝、灯珠、文字或反光点当成圆。negative
数据集必须包含没有目标和相似干扰物，并统计误检。

## 5. 为什么总体 95% 可能仍然不能比赛

假设正常光照 900 张全对，逆光 100 张只对 50 张，总体仍有 95%。但比赛恰好是逆光时，算法
会有一半时间失败。因此报告必须同时写总体结果和最差场景切片。

## 6. 初学者调试顺序

1. 先看 `raw`，确认有没有过曝、过暗、失焦或运动模糊。
2. 再看相机曝光/增益和 ROI，不要立即改轮廓参数。
3. 用固定图片一次只改一个参数，比较误检、漏检和中心误差。
4. 把失败帧加入对应场景切片，不覆盖原图。
5. 重新跑全部回归，防止修好暗场却破坏正常光照。
6. 条件恢复后检查 valid、置信度和跟踪状态是否正常恢复。

## 7. 当前工程做到了哪一步

工程现在只有鲁棒性规范和文件回放底座，没有矩形、圆或光斑正式算法，也没有真实相机扰动
数据。当前能说的是“以后知道怎样验证”，不能说“识别已经鲁棒”。
