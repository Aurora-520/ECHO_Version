# MaixCAM 首次连接：先证明链路，再运行视觉

## 1. 为什么电脑会出现两张“网卡”

MaixCAM 通过一根 USB-C 线模拟网络设备。Windows 11 可能同时安装 NCM 和 RNDIS 两种驱动，
所以会看到两个网段。这不是两台 MaixCAM，也不一定是驱动冲突。

本次电脑地址是 `10.5.66.100` 和 `10.5.67.100`，设备地址按官方规则把最后一段改成 `.1`，即
`10.5.66.1` 和 `10.5.67.1`。两条都能访问 SSH，自动化时选一条稳定链路即可。

## 2. 端口通不等于设备已经通过测试

- 22 开放：SSH 服务可达。
- 7899 开放：MaixVision 服务在监听。
- 能登录：账户和网络正常。
- 能实时抓图：还需要相机所有权和 Camera API 通过。
- 能比赛：还需要曝光、延迟、鲁棒性、UART 和整车验证。

这些是不同验证层级，不能混写。

## 3. 为什么不能随便 `import maix`

普通 Python 包通常只是提供函数，但 MaixPy 4.12.5 的顶层导入会执行初始化，并注册默认通信
监听，本次实机直接显示 UART0 被打开。版本探测这种本应只读的操作因此可能抢占串口。

读取版本应优先看：

```text
/boot/ver
MaixPy-*.dist-info/METADATA
maix/version.py
```

正式程序仍可以导入 MaixPy，但必须把它放在平台适配层，并保证进程是相机/UART 的唯一所有者。

## 4. 为什么看不到 `/dev/video0`

树莓派/USB 相机经常使用 V4L2 的 `/dev/video0`。MaixCAM 使用 CVI/Sophgo 多媒体中间件，本次
设备看到的是 `cvi-vi`、`cvi-vpss`、`cvi-isp` 等节点，没有普通 V4L video 节点。

因此不能把树莓派 OpenCV `VideoCapture(0)` 原样搬到 MaixCAM，必须使用 MaixPy Camera API 的
薄适配器。

## 5. 资源数字怎么看

本次 Linux 只看到一个 CPU 核、约 128MiB 内存且无 swap。`free` 很小不一定马上 OOM，因为
Linux 会把空闲内存用于缓存；应关注 `MemAvailable`。但录像、模型、overlay 和日志同时运行仍然
很容易耗尽内存，所以必须使用有界队列和固定内存预算。

发现时约 55°C 只是一个快照，不能代表连续运行温度。正式结论需要记录环境、负载和时间曲线。

## 6. 设备时间为什么重要

设备当前日期比电脑落后近一个月。单调时间仍可计算一次启动内的延迟，但墙钟错误会污染文件名、
数据集时间和跨设备日志。正式采集前要先修正时间来源，并记录校时前后证据。

## 7. 下一次连接的固定顺序

1. 运行发现脚本并检查 `DiscoveryErrors`。
2. 确认 NCM/RNDIS 的主机地址和 `.1` 设备地址。
3. 检查 SSH 22，不立即刷系统。
4. 读取镜像、版本、存储、内存、温度和当前进程。
5. 检查哪个应用正在拥有相机和 UART。
6. 正常释放资源后才启动我们的采集程序。
