# ECHO_Vision 鲁棒性与赛场环境验收规范

本文件是所有视觉算法、相机配置、Pipeline 和任务状态机的横向验收门禁。鲁棒性不是“永不
失败”，而是：在声明的工作范围内保持性能，接近边界时能被观测，超出范围时拒绝错误结果，
并在条件恢复后可预测地恢复。

## 1. 必须覆盖的扰动

| 类别 | 典型赛场变化 | 必须记录的证据 |
| --- | --- | --- |
| 光照强度 | 暗场、强光、局部亮暗、灯光开关 | 原图、亮度统计、曝光、增益、成功/失败 |
| 光谱与颜色 | 冷暖色温、彩色环境光、白平衡漂移 | 白平衡模式/参数、颜色统计、分场景指标 |
| 光照方向 | 逆光、侧光、硬阴影、目标自身阴影 | 场景标签、失败帧、误检/漏检 |
| 反光与饱和 | 亚克力、金属、白纸高光、灯珠光晕 | 高亮裁剪比例、轮廓/中心偏差 |
| 频闪与瞬态 | LED/荧光灯频闪、自动曝光收敛 | 帧序列、曝光变化、恢复时间 |
| 运动与成像 | 运动模糊、抖动、失焦、脏镜头 | 模糊度/清晰度指标、速度、失败比例 |
| 几何变化 | 距离、尺度、旋转、透视、画面边缘 | 位姿/区域标签、坐标误差、最差切片 |
| 遮挡与背景 | 部分遮挡、相似干扰物、杂乱背景 | negative 集、误检率、拒绝行为 |
| 设备与性能 | 相机个体、镜头、温度、降频、丢帧 | 设备 ID、版本、温度、FPS、结果年龄 |
| 故障与恢复 | 相机断开、坏帧、配置错误、通信失联 | fault、valid、恢复时间、MCU 策略 |

没有照度计、色温计或可靠设备读数时，使用可复现的场景 ID、灯具位置、相机参数和原始图像，
不得编造 lux、色温或曝光物理量。

## 2. 分层控制策略

### 采集层

- CameraSource 是唯一相机所有者，负责曝光、增益、白平衡、分辨率和帧率的原子配置。
- `Frame.metadata` 应逐步加入实际可读取的 `exposure_us`、`gain_db`、白平衡、传感器/镜头 ID、
  分辨率和采集状态；设备不支持的字段明确为 unknown。
- 自动曝光用于找范围时必须记录收敛过程；正式任务可选择锁定或受限自动曝光，但必须经 A/B
  数据决定，不能凭习惯冻结。
- V1 应建立原图与相机参数同步保存，避免只保存叠加图而无法复现。

### 算法与估计层

- 纯算法只接收图像、参数和必要状态，不直接修改相机配置。
- 优先使用几何约束、相对量、局部对比度和多特征一致性，避免只依赖单个固定灰度阈值。
- 自适应阈值、归一化和颜色空间转换必须有上下限、诊断量和固定数据回归，禁止无界在线自调。
- 跟踪只能提升连续性，不能把长期未检测伪装成当前检测；预测结果必须携带来源、年龄和置信度。

### Pipeline 与输出层

- Pipeline 在发布结果前检查图像质量、检测置信度、几何合理性、时间一致性和结果年龄。
- 不满足门禁时发布 `valid=false` 或故障标志，保留最近合法目标只能由明确的任务策略决定。
- 失败帧、边界帧和低置信度帧进入旁路保存，不允许阻塞采集/推理。
- MCU 必须根据 valid、confidence、age、fault 和通信状态执行零输出、停止或受限保持策略。

## 3. 数据集切片

每项能力至少维护以下互斥用途：

- `train`：只用于模型训练；传统视觉可为空。
- `dev`：用于调参数和设计规则。
- `test`：冻结后只用于验收，不根据结果继续人工挑参数。
- `field`：真实台架/整车/赛场采集，只追加版本，不覆盖原始文件。

每次评测至少包含：

- nominal：预期正常环境。
- boundary：工作范围上下边界。
- adverse：逆光、阴影、反光、模糊、遮挡等恶劣条件。
- negative：没有目标或存在相似干扰物。
- recovery：从遮挡、断开、曝光突变或坏帧恢复。

数据集索引必须记录设备、镜头、分辨率、帧率、曝光/增益、场景标签、拆分、标签版本和 manifest
SHA-256。大文件保存在 Git 外，只提交元数据和清单哈希。

## 4. 鲁棒性验证等级

| 等级 | 含义 | 可声明内容 |
| --- | --- | --- |
| R0 | 未评估 | 只能说接口/占位存在 |
| R1 | 宿主机固定数据切片通过 | 可说离线验证，不可说硬件可用 |
| R2 | 真实相机台架扰动测试通过 | 可说该相机/配置在记录范围内通过 |
| R3 | 目标平台连续运行与故障恢复通过 | 可进入任务级联调 |
| R4 | 整车同场景、压力和赛场演练通过 | 可作为比赛候选基线 |

鲁棒性等级必须绑定 capability 版本、配置哈希、数据集版本、平台和验证日期，不能从一个算法
自动继承到另一个算法，也不能从树莓派结果直接继承到 MaixCAM。

## 5. 能力升级门禁

算法或模块标记 `reusable` 前必须：

1. 在能力文档声明输入、输出、正常/退化/拒绝范围和安全回退。
2. 在调参前冻结 test 集和任务级验收阈值；阈值尚未知时明确 `TBD`，不得事后挑标准。
3. 分别报告 nominal、boundary、adverse、negative、recovery 指标和最差切片。
4. 报告 precision、recall、坐标误差、valid 拒绝率、恢复时间、FPS、p50/p95、drop 和结果年龄。
5. 保存失败样本和误检样本，并在 debuglog 中解释主要失败模式。
6. 证明日志、录像、HTTP 和失败帧旁路不会使高频路径超出预算。
7. 在适用的真实目标平台至少达到 R2；只有 R1 时状态最多为 `host-validated`。

## 6. 赛场调参与回退

- 预先准备版本化配置档，例如 nominal、dim、bright，不在赛场临时散改源码。
- 每次只改变一类变量，先保存原图、当前配置和状态，再判断曝光、算法还是坐标/控制问题。
- 参数更新在帧边界原子生效，保留上一版可一键回退配置。
- 现场调试先看 raw、overlay、status 和失败帧；只看叠加图无法判断原始曝光是否已经失真。
- 条件超出已验证范围时优先安全拒绝和切换备用策略，不得临时关闭 valid/age/confidence 门禁。

## 7. 分阶段落实

- V1：完成相机曝光、增益、白平衡、频闪、清晰度和参数元数据采集。
- V2：为矩形、圆、光斑、黑线等建立固定鲁棒性数据切片和离线评测。
- V3：验证失联、坏帧、过期、低置信度和恢复时 MCU 的安全策略。
- V4：验证任务状态切换、跟踪和多帧估计在扰动下不会持续输出假目标。
- V5：完成同场景 A/B、连续运行、整车运动、温度和赛场灯光演练，达到 R4 才能冻结比赛基线。
