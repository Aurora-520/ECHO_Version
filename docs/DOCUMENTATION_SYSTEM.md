# ECHO_Vision 文档与知识传承体系

本文件定义工程事实、调试经验、学习材料和可复用模块如何长期保存。目标是让新的聊天、队友或
赛场临时接手者不依赖口头记忆，能够判断当前状态、理解过去的问题并安全复用已有能力。

## 1. 五类记录不能互相替代

| 类型 | 位置 | 回答的问题 |
| --- | --- | --- |
| 当前状态 | `PROJECT_STATUS.md` / `CURRENT_HANDOFF.md` | 现在做到哪里、下一步是什么 |
| 工作日志 | `docs/worklogs` | 这次工作修改了什么、证据是什么 |
| 调试日志 | `docs/debuglogs` | 出现什么问题、如何定位、怎样修复和避免复发 |
| 学习日志 | `docs/learning` | 用初学者能理解的方式解释原理和排障方法 |
| 能力目录 | `docs/CAPABILITY_CATALOG.md` / `docs/capabilities` | 已调模块怎样快速、安全地复用 |

不能把聊天记录当文档，不能用 worklog 的一句“已修复”代替完整 debuglog，也不能用源码注释
代替面向初学者的学习材料。

## 2. 新任务和聊天交接的强制阅读

每个新任务、重开聊天或更换执行者时，必须先读取：

1. `AGENTS.md`
2. `docs/PROJECT_STATUS.md`
3. `docs/CURRENT_HANDOFF.md`
4. 本文件
5. `docs/CURRENT_WORKFLOW.md` 和当前 Phase 文档
6. `docs/CAPABILITY_CATALOG.md`
7. `docs/debuglogs/README.md` 和当前阶段/模块相关的最近调试日志
8. `docs/learning/README.md` 和本任务相关的学习日志
9. Git 状态和本次任务直接相关源码

不要求每次加载所有历史日志，但必须读取索引，并根据索引读取当前模块相关记录。不能因为
聊天中“记得大概”而跳过。

## 3. 阶段完成的文档门禁

一个阶段只有同时满足以下条件，才能写“阶段完成”并创建阶段 tag：

- Phase 文档中的设计、实现和验证门禁通过。
- `PROJECT_STATUS.md` 和 `CURRENT_HANDOFF.md` 已更新。
- 本阶段 worklog 已记录修改、硬件状态和验证证据。
- 本阶段 debuglog 已记录所有重要失败、根因、修复、验证和预防措施。
- 新知识或新排障方法已写入 learning 日志。
- 已调模块已登记到能力目录，并有复用说明、边界、示例和回归测试入口。
- 所有未执行硬件测试仍明确标记 `deferred`。

## 4. 模块“可复用完成”的门禁

模块只有具备以下内容，才能登记为 `reusable`：

- 明确目的、输入、输出、坐标系、单位和数据所有权。
- 依赖、配置参数、默认值、范围和版本。
- 是否打开设备、是否阻塞、队列容量、超时和线程安全说明。
- 无目标、低置信度、过期、异常和资源不足时的行为。
- 最小使用示例和接入 Pipeline/Mission 的步骤。
- 单元测试、固定数据集回放和适用平台的真实验证层级。
- 性能基线和资源预算；没有实测时明确写未知。
- 已知限制、不能使用的场景和安全回退方法。
- 来源 commit/tag，以及配置、模型或标定哈希。

缺少其中任何关键内容时，状态只能是 `experimental`、`host-validated` 或 `deferred`，不能写
“比赛可直接使用”。

## 5. 记录原则

- 问题日志记录事实链：现象 -> 假设 -> A/B 证据 -> 根因 -> 修复 -> 回归 -> 预防。
- 学习日志先解释“为什么”，再给术语和代码位置，不假设读者已有视觉基础。
- 能力文档只提供稳定入口，不要求比赛 Mission 复制内部实现。
- 旧日志不重写历史；发现新结论时追加勘误或新文件。
- 原始大日志和视频保存在 Git 外，文档记录路径、时间、版本和 SHA-256。

## 6. 每次任务结束的最小检查

- 本次修改、验证和未执行项已写入 worklog。
- 本次出现的重要问题已写入 debuglog；没有运行错误时也记录环境限制和主动预防的风险。
- 新增知识已转写为初学者可理解的 learning 内容，或明确说明本次没有新增知识。
- 调整过的模块已更新能力状态、接入方法、回归入口和限制。
- `PROJECT_STATUS.md` 与 `CURRENT_HANDOFF.md` 能让下一次聊天不依赖本次对话继续工作。
