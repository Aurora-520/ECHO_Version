# 工作日志规则

有源码、配置、长期文档、硬件测试或关键决定的会话必须新建：

```text
YYYY-MM-DD_<short-topic>.md
```

必填章节：目标与范围、开始状态、调查/修改、硬件状态、验证、问题与判断、风险与下一步、
结束状态。写“通过”必须有对应命令或硬件证据；没有执行的项目写“未执行”或 `deferred`。

大型原始日志、视频、数据集、权重和聊天记录不进入 Git。worklog 只保存摘要、稳定路径、版本、
哈希和必要的小型证据。旧日志不因当前状态变化而重写。

## 与长期知识文档的分工

- worklog 记录一次任务的范围、修改和验证，不替代完整问题复盘。
- 重要问题的现象、证据、根因、修复和预防写入 `docs/debuglogs`。
- 面向无视觉基础队员的原理和排障方法写入 `docs/learning`。
- 已调模块的稳定接口、配置、故障行为、回归入口和复用步骤写入 `docs/capabilities`。
- 每个阶段结束前按 `docs/DOCUMENTATION_SYSTEM.md` 检查以上记录是否完整。
