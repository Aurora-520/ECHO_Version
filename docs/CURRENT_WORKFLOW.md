# ECHO_Vision 当前工作流程

## 1. 长期阶段

```text
V0 工具链、规则、结构化类型、文件源和可运行回放
-> V1 MaixCAM/树莓派采集、曝光、标定、时间戳和录像
-> V2 基础视觉算法与离线数据集评测
-> V3 视觉-MCU 协议、UART 实测和故障恢复
-> V4 任务算法、状态输出、跟踪和实时优化
-> V5 整车联调、压力测试、故障演练和赛场验收
```

不得因为赛题未确定而跳过 V0/V1，也不得在 V0 堆入大量具体赛题状态机。

## 2. 每阶段固定闭环

1. 设计边界：范围、非目标、所有权、时间约束和验收标准。
2. 独立实现：只做当前阶段，不混入下一阶段硬件和 Mission。
3. 构建与静态检查：记录环境、版本、错误和警告。
4. 离线回放：使用固定数据集和配置，保存机器可读摘要。
5. 单模块实测：相机、UART、调试服务分别验证。
6. 故障测试：断开、超时、坏帧、队列满、低置信度、过期和恢复。
7. 连续运行：无断点，记录时长、FPS、p50/p95、丢帧、资源和温度。
8. 文档闭环：更新状态、交接、Phase 和 worklog；完成本阶段 debuglog；新增知识写入 learning；
   已调模块更新 capability 文档和目录。
9. 提交与标签：全部门禁通过后才创建阶段 commit 和 annotated tag。
10. 下一阶段：只能从已验收的新基线开始。

## 3. 开工流程

按 `AGENTS.md` 和 `docs/DOCUMENTATION_SYSTEM.md` 的读取顺序执行。必须先读能力、调试和学习
索引，再按本任务模块读取相关条目。修改 dirty 文件前先读 diff；大数据、录像和权重先建立
元数据和哈希，不直接加入 Git。

## 4. 阶段与模块复用门禁

- 每个阶段至少有一份阶段调试总结；重要模块问题另建 debuglog。
- 每个重要问题记录现象、证据、根因、修复、验证、预防和真实任务影响。
- 每个新原理或排障方法转写为初学者可理解的 learning 内容。
- 已调模块只有满足 `docs/DOCUMENTATION_SYSTEM.md` 的接口、所有权、配置、故障、示例、测试、
  性能、限制和版本要求后，才可标记 `reusable`。
- 新 Mission 通过稳定接口和集中配置组合能力，不复制内部实现，不新增相机或 UART 所有者。
- 文档和验证证据缺失时，阶段不能完成，能力状态不能升级。

## 5. 平台决策流程

MaixCAM Pro 和树莓派使用相同接口与同一批回放数据。平台选择必须以同场景 A/B 数据为依据：

- 支持的相机、曝光和分辨率。
- FPS 与端到端 p50/p95 延迟。
- 精度、漏检、误检和结果年龄。
- CPU/NPU、内存、温度和连续运行稳定性。
- 部署、恢复、赛场排障和备用成本。

没有 A/B 证据前只写“暂定”，不写“最终平台”。

## 6. 硬件门禁

- 第一次连接 MaixCAM：只读发现 USB 网卡、IP、系统、存储、进程和串口。
- 普通开发：SSH/SCP 上传应用、模型和配置，不刷系统。
- 系统烧录：必须走 `docs/BENCH_WORKFLOW.md` 的破坏性门禁并等待用户现场确认。
- UART：确认 3.3 V TTL、TX/RX 交叉、可靠共地后才连接。
- 整车动作：视觉只输出结果，MCU 安全门和唯一执行器写入者必须先通过。
