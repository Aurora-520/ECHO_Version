# 调试日志规则

`docs/debuglogs` 保存工程中出现过的重要问题及其完整解决过程。每个阶段结束前必须至少有一份
阶段调试总结；同一模块有独立复杂问题时，应另建模块调试日志。

文件名：

```text
YYYY-MM-DD_<phase-or-module>_<topic>.md
```

## 必填内容

```markdown
# <日期> <问题或阶段>

## 背景与影响
## 现象
## 初始假设
## 调查与 A/B 证据
## 根因
## 修复
## 验证
## 如何避免复发
## 对真实视觉任务的影响
## 可复用结论
## 仍未解决
```

## 强制规则

- 不能只写“改参数后好了”，必须记录改变了哪个变量、为什么和验证条件。
- 编译、回放、设备部署、UART 和整机证据必须分层记录。
- 偶发问题必须记录出现频率、触发条件和不能证明的部分。
- 修复后必须说明未来怎样通过接口、配置、测试或诊断防止同类问题。
- 阶段没有出现运行错误时，也要记录环境限制、未执行门禁和设计中主动避免的风险。
- 问题属于临时环境时，同样记录，防止新聊天把环境失败误判成代码失败。

## 与其他文档的关系

- worklog：一次工作整体发生了什么。
- debuglog：一个问题怎样被证明和解决。
- learning：把问题背后的知识讲给初学者。
- capability：修复后的模块怎样稳定复用。
