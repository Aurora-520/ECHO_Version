# 2026-07-15 鲁棒性治理补强

## 背景与影响

正式赛场的灯具、阴影、反光、曝光收敛、运动和背景都会与实验室不同。若工程只记录“某张图
识别成功”，算法可能在比赛现场持续输出错误坐标，并把问题传递给 MCU。

## 现象

原工程风险清单提到光照变化，但没有统一扰动矩阵、最差切片、验证等级和能力升级门禁。数据集
索引也没有区分正常、恶劣、负样本和恢复场景。

## 初始假设

为每个算法单独增加几个不同亮度样本即可说明鲁棒性。

## 调查与 A/B 证据

仅改变亮度无法覆盖色温、阴影、逆光、反光、频闪、运动模糊、失焦、尺度、遮挡、相似干扰物、
相机个体和温度降频。总体平均指标还可能掩盖某个场景切片完全失效，因此需要统一的跨层门禁。

## 根因

原 V0 重点是数据流和安全协议，还没有目标算法与正式数据集，因此没有把赛场环境变化建模为
架构级输入和验收维度。

## 修复

- 新增 `docs/ROBUSTNESS.md`，定义扰动、分层控制、数据切片、R0-R4 等级和赛场回退。
- 更新 AGENTS、架构、红线、阶段流程、能力门禁、指标、风险和数据集元数据。
- 明确鲁棒性失败时使用 valid/confidence/age/fault 安全拒绝，禁止输出猜测坐标。

## 验证

- `git diff --check` 在暂存前检查通过。
- Python 3.12.13 且 `PYTHONPATH=src` 时，原 V0 单元测试 14/14 通过。
- 扩展后的 `datasets/index.example.csv` 使用标准库 `csv.DictReader` 解析通过，共 16 列、1 行。
- 本次只验证文档、索引和原 V0 单元测试，没有正式检测算法和真实相机数据，目标识别鲁棒性
  状态保持“未执行”。

## 如何避免复发

任何新算法在调参前建立冻结 test 集，按 nominal、boundary、adverse、negative、recovery 报告
指标和最差切片；能力文档绑定数据集、配置、平台和 R 等级。

## 对真实视觉任务的影响

矩形、圆、光斑、黑线、棋盘和运动目标都使用同一鲁棒性框架。条件超出已验证范围时，视觉明确
拒绝结果，MCU 按安全策略停止或受限保持，而不是根据假坐标驱动云台/底盘。

## 可复用结论

鲁棒性由“数据覆盖 + 可观测元数据 + 分层质量门禁 + 安全拒绝 + 恢复测试”共同构成，不能靠
单个自适应阈值解决。

## 仍未解决

- 相机型号、曝光 API 和真实赛场照明范围未知。
- 尚无矩形/圆/光斑正式数据集和任务级数值门槛。
- R2-R4 必须在真实相机、目标平台和整车上测试。
